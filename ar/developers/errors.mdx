عند فشل الطلب، تُعيد Integration API رمز حالة HTTP قياسيًا وجسمًا بصيغة JSON يحتوي
على `message` مقروء يصف ما حدث.

```json
{
  "message": "Idempotency-Key header is required."
}
```

تحقّق دائمًا من رمز الحالة أولًا، ثم اقرأ `message` للاطّلاع على التفاصيل.

## رموز الحالة الشائعة

| الحالة | المعنى | كيفية المعالجة |
|---|---|---|
| **400** Bad Request | فشل التحقّق، أو غياب ترويسة `Idempotency-Key` المطلوبة. | راجع `message`. أضِف ترويسة `Idempotency-Key` وصحّح أي حقول غير صالحة. |
| **401** Unauthorized | ترويسة `X-Api-Key` مفقودة أو المفتاح غير صالح أو مُلغًى. | أرسل مفتاحًا صالحًا في `X-Api-Key`. إن كان مُلغًى، أنشئ مفتاحًا جديدًا. |
| **402** Payment Required | بلغت حصّتك الشهرية من المستندات. | انتظر إعادة تعيين الحصّة أو ارتقِ بخطتك. |
| **404** Not Found | نقطة النهاية أو المورد المُشار إليه غير موجود. | تحقّق من العنوان ومعرّف المورد ومن أن شركة المفتاح تملكه. |
| **422** Unprocessable Entity | فشل المستند في تحقّق PINT-AE (schematron). | اقرأ `message` لمعرفة القاعدة المخالفة وصحّح بيانات الفاتورة (مثل غياب `exemptionReasonCode` على سطر معفى). |

<Note>
**الفرق بين 400 و422.** يعني **400** أن الطلب نفسه مشوّه — JSON غير صالح أو ترويسة
مفقودة أو حقل لا يجتاز التحقّق. أما **422** فيعني أن الطلب سليم البنية لكن الفاتورة
الإلكترونية الناتجة لا تجتاز قواعد PINT-AE الإماراتية.
</Note>

## التحقّق والحصّة

- ترويسة **`Idempotency-Key`** إلزامية في كل `POST`؛ وإغفالها هو السبب الأكثر
  شيوعًا لـ **400**. راجع [معرّف عدم التكرار](/ar/developers/idempotency).
- يكون `exemptionReasonCode` **مطلوبًا** على أي سطر يكون فيه `taxCategoryCode`
  بقيمة `E`؛ وإغفاله يُطلق خطأ تحقّق.
- يتعلّق **402** بخطتك لا بطلبك — فالطلب كان صالحًا. ولن تُجدي إعادة المحاولة حتى
  تُعاد الحصّة.

## معالجة الأخطاء في الكود

- أعِد محاولة الأخطاء **العابرة** (أخطاء الشبكة والمهل المنتهية) بنفس
  `Idempotency-Key` كي لا تُكرّر مستندًا أبدًا.
- **لا** تُعِد محاولة **400/401/402/404/422** دون تفكير — فهذه تحتاج إلى إصلاح
  (تصحيح الحمولة أو المفتاح أو خطتك)، لا إلى محاولة أخرى.
